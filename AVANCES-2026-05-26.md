## Decisiones de diseño

### Tipografías
| Variable | Fuente | Uso |
|---|---|---|
| `--font-headline` | Cormorant Garamond | Títulos, H1-H3, subtítulos |
| `--font-body` | Inter | Cuerpo de texto, UI funcional |
| `--font-bitacora` | DM Mono | Etiquetas, metadatos, voz `//` |

Cormorant aporta elegancia editorial + itálicas reales que Inter no tiene.
DM Mono ancla la voz de bitácora con un tono analógico / de log de navegación.

### Paleta warm
- `--color-warm: #d4a574` — dorado pergamino, color de acento principal
- Reemplaza el uso del tertiary azul en casi todos los elementos de UI activa
- Los tres pilares (arte / ciencia / consciencia) conservan sus propios tokens

### Voz de bitácora (`//`)
Prefijo `//` en todos los textos de metadato / etiqueta. Evoca registro de
navegación, log de expedición. Aparece en eyebrows, countLabels, headers de
footer, selector de idioma, separadores, chips de episodio.

No es decorativo — es parte del sistema de voz del proyecto.

### Trailer en EpisodeList
El trailer (adelanto) no es parte de la playlist de YouTube. Embed sin `?list=`.
Decisión: si `featuredVideoId` existe → aparece primero como ítem destacado,
el episodio 01 pierde el estado activo inicial. Backwards-compatible: sin
`featuredVideoId` el componente funciona exactamente igual que antes.

### Decisiones de diseño

#### Patrón inline style para colores de pilar
Las utilities arbitrarias de Tailwind 4 con sintaxis `bg-[color:--var]/opacidad`
no funcionan en este proyecto (generan CSS inválido sin envolver la variable
en `var()`). Las utilities nativas `bg-pillar-arte/10` tampoco funcionan
(no se compilan). Solución adoptada: inline style con `rgb(from var(--color-pillar-X) r g b / alpha)`.

Este patrón se aplica en PillarBadge y en los íconos de Tres Pilares. Es
reutilizable para cualquier sección futura que necesite colores dinámicos
por pilar o por token CSS.

#### Voz del Newsletter diferenciada del CTA del Header
- Header CTA: `A bordo` / `Get on board` → lleva a /contacto.
- Newsletter button: `Embarcate` / `Embark` → suscripción al newsletter.
Misma metáfora marítima, distintas acciones, distintos textos.

### Decisiones de diseño

#### Glifos tipográficos como patrón sistemático
Reemplazo de `material-symbols-outlined` por glifos Unicode renderizados en `font-headline` (Cormorant Garamond). Ya consolidado en Vision Quote del index, ahora aplicado también en Mission (✦) y Vision (⊙) de institucional. Es un patrón estable del sistema, no decisión one-off. Razón: las itálicas y glifos de Cormorant Garamond son material de diseño desperdiciado si se usan icon fonts.

#### Botones circulares de redes — un solo lenguaje para el sitio
Los círculos `rounded-full border w-X h-X hover:border-warm hover:-translate-y-0.5` son el patrón de "botón circular de red social" en todo el sitio. El Footer los usa a w-10 h-10. Descubrimos en este sprint que los valores del CMS (content collection) sobreescriben los fallbacks del código. Si el `.md` tiene `youtube_url: "..."`, ese valor gana siempre, aunque el fallback del componente esté actualizado. Lección: refactor visual y refactor de contenido son trabajos distintos. Para cambios de URL/texto, tocar el CMS, no el código.

### Objetivo
Integrar Keystatic CMS al proyecto para editar el contenido de `src/content/projects/` desde un panel visual.

### Lo que se hizo

**1. Merge a main**

**2. Instalación de dependencias**
```bash
npm install @keystatic/astro @keystatic/core react react-dom @astrojs/react
npm audit fix  # resolvió 1 vulnerabilidad alta en devalue (Svelte/transitiva)
```

**3. Configuración inicial**
- `astro.config.mjs`: se agregaron `react()` y `keystatic()` al array `integrations`.
- La integración `keystatic()` registra automáticamente las rutas `/keystatic/[...params]` y `/api/keystatic/[...params]` — no se deben crear manualmente (error de ruta duplicada confirmado y corregido).
- `keystatic.config.mjs` creado en la raíz con schema completo para la colección `projects`.

### El problema encontrado

Keystatic no encontraba los archivos `.md` existentes. El panel cargaba pero la colección aparecía vacía (redirigía a `/create`).

**Causa raíz** (encontrada en el source de `@keystatic/core`):

```js
// read-local-bc5599c1.node.js, línea 652
const dataLocation = path.endsWith('/') ? 'index' : 'outer';

// index-360e5820.node.js, línea 645
return formatInfo.contentField ? formatInfo.contentField.contentExtension : '.' + formatInfo.data;
```

Sin `format` explícito, Keystatic usa `data: 'yaml'` por defecto → busca archivos `.yaml`, no `.md`. Nuestros archivos son `.md`.

Con `format: { contentField: 'algo' }` la extensión cambia a `.md`, pero `contentField` requiere un campo de tipo editor (`fields.document`, `fields.markdoc`) — no `fields.text`. Usar `fields.text` como contentField causa un crash silencioso en React (pantalla en blanco).

### La solución decidida

Convertir `src/content/projects/` de `.md` a `.yaml`. Keystatic con `format: { data: 'yaml' }` (sin `contentField`) busca exactamente archivos `.yaml` con datos YAML puros — que es el formato real de estos archivos (son frontmatter sin cuerpo markdown).

### Estado actual de keystatic.config.mjs (pendiente limpiar)

El config tiene temporalmente:
```js
format: { data: 'yaml', contentField: 'content' },  // ← borrar contentField
// ...
content: fields.text({ label: 'Notas internas', multiline: true }),  // ← borrar este campo
```
Ambos se eliminan al hacer la migración.

### Próximo paso

1. Renombrar todos los archivos en `src/content/projects/` de `.md` a `.yaml`
2. Limpiar `keystatic.config.mjs`: dejar `format: { data: 'yaml' }`, eliminar campo `content`
3. Actualizar cualquier import de Astro que referencie los archivos por extensión (si los hay)
4. Verificar que el panel liste los 7 proyectos y que se puedan editar sin romper el contenido
